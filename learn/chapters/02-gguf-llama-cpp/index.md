## 0. 第1章とのつながり（GPUで訓練 → CPUで配布・推論）

[第1章](../01-bitsandbytes-qlora/index.md)は、**GPU の上でモデルを量子化しながら微調整する**話でした（bitsandbytes/NF4・QLoRA）。あれは「学習のための量子化」です。

第2章は、その先——**できあがったモデルを、配って・手元のPC（CPU）で動かす**話です。そのための定番が **GGUF 形式**と、それを動かす **`llama.cpp`** です。

> ざっくり：**第1章＝鍛える（GPU）／第2章＝持ち運んで動かす（CPU）**。

## 1. GGUF とは何か（なぜ CPU 推論に向くのか）

**GGUF（GGML Universal Format）** は、`llama.cpp` が使う**単一ファイルのモデル形式**です。重みだけでなく、トークナイザやメタデータ（アーキテクチャ情報など）も**1つの `.gguf` ファイルに同梱**します。

- **単一ファイル**：1つ落とせば動く。配布・共有が簡単
- **量子化前提**：4bit などに圧縮した重みをそのまま保持できる → **ファイルが小さい**
- **CPU で動く**：`llama.cpp` は C/C++ 実装で、GPU が無くても（CPUだけでも）推論できる

第1章の「実行時に4bitでロードする（bitsandbytes）」とは違い、GGUF は**最初から量子化された状態でファイルに焼き込む**のがポイントです。

## 2. llama.cpp とは何か

**`llama.cpp`**（リポジトリは `ggml-org/llama.cpp`）は、GGUF モデルを動かすための軽量な推論エンジンです。

- **`convert_hf_to_gguf.py`**：Hugging Face のモデルを GGUF（16bit）に**変換**するスクリプト
- **`llama-quantize`**：GGUF を 4bit などに**量子化**するツール
- **`llama-completion` / `llama-cli`**：GGUF を使って**推論**するプログラム

この章のハンズオンでは、この3つを順に使って「変換 → 量子化 → 推論」を一気に通します。

## 3. 量子化タイプの読み方（Q4_K_M など）

GGUF の量子化タイプは `Q4_K_M` のような名前で表します。記号の意味が分かると選べるようになります。

| 記号 | 意味 |
| :-- | :-- |
| **Q4** | 重みを **4bit** で表す（数字が小さいほど軽い・粗い） |
| **_K** | **K-quant**（ブロック単位で精度を最適化した新しい方式） |
| **_M** | サイズの目安（**S**mall / **M**edium / **L**arge。同じbit数でも質と容量が変わる） |

代表的な選択肢の位置づけ（ざっくり）：

| タイプ | 傾向 | こんなとき |
| :-- | :-- | :-- |
| `Q4_K_M` | **軽さと質のバランス型・定番** | まず迷ったらこれ |
| `Q5_K_M` | やや重いが質が高い | 質を少し優先したい |
| `Q8_0` | ほぼ無劣化だが大きい | 質を最優先・容量に余裕 |
| `Q4_0` | 最軽量級だが質は粗い | とにかく小さく |

!!! note "詳しい仕様は公式へ"
    使えるタイプの全一覧と各定義は、本文ではなく [llama.cpp quantize README](references.md) を正としてください（バージョンで増減します）。より高品質な量子化のための **imatrix**（キャリブレーション）という発展手法もありますが、本章では名前だけ。

## 4. 第1章（bitsandbytes / NF4）との使い分け

同じ「4bit量子化」でも、目的が違います。混同しないように。

| | 第1章：bitsandbytes / NF4 | 第2章：GGUF / llama.cpp |
| :-- | :-- | :-- |
| 何のため | **学習（微調整）向け**の実行時量子化 | **配布・推論向け**のファイル形式 |
| 動く場所 | 基本 **GPU** | **CPU** でも動く |
| 形 | 実行時にメモリ上で4bit化 | あらかじめ4bitに焼いた `.gguf` ファイル |
| 代表ツール | transformers + bitsandbytes + peft | llama.cpp（convert / quantize / completion） |

- **微調整したい** → 第1章（QLoRA）
- **軽く・速く・CPUで動かして配りたい** → 第2章（GGUF）
- **GPU で精度を保ったまま軽くしたい** → [第3章（GPTQ）](../03-gptq/index.md)

## トラブルシュート（実行中に詰まったら）

ハンズオン中によく出る問題と対処です。**実行しながら別タブで参照**してください。

| 症状 | 主な原因 | 対処 |
| :-- | :-- | :-- |
| バイナリのDLに失敗する | ネットワーク／タグ名の誤り | タグ `b9743` の `*-bin-ubuntu-x64.tar.gz` を使う。失敗時はランタイムを再起動し**セットアップから流し直す**（Run all） |
| `convert_hf_to_gguf.py` が見つからない | バージョンでスクリプトの場所が変わる | パスを決め打ちせず、Notebook のように `find` で探してから実行する |
| `llama-quantize` / 実行ファイルが無い | 配布物の構成が変わることがある | こちらも `find` で実パスを解決してから使う |
| 推論が同じ語を繰り返す/空白だらけ | base モデルを `temp 0` で長く生成すると起きやすい | `-n`（生成トークン数）に上限を付ける。デモはこれで十分 |
| メモリ不足で落ちる | 変換時に元モデルを読み込むため | より小さいモデル（例：`SmolLM2-360M`）に変える |

!!! warning "ローカル（自分のPC）で動かす場合の注意"
    このNotebookは **Colab の CPU ランタイム（隔離環境）** での実行を前提にしています。手元のPCで直接動かすとモデル変換でメモリを多く使うため、非力な機種では重くなることがあります。まずは Colab を推奨します。
