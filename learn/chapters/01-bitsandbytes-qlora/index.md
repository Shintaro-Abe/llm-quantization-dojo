## 1. なぜ量子化するのか（VRAM とコスト）

LLM の重みは通常 16bit（fp16/bf16）で保持されます。これを **4bit** に落とすと、重みのメモリ使用量はおよそ **1/4** になります。

- **効果**：手元の小さいGPU（例：無料Colabの T4・約15GB）にも、より大きなモデルが載る／推論や微調整が現実的になる
- **コスト**：高価なGPUを長時間借りなくても学習・検証ができる
- **トレードオフ**：精度はわずかに落ちうる。だが多くの実務タスクでは**実用上問題ない範囲**に収まることが多い

## 2. NF4 とは何か

**NF4（4-bit NormalFloat）** は、量子化のための4bitデータ型です。LLMの重みは「0付近に集中する正規分布に近い」性質を持つため、NF4はその分布に合わせて数値の刻みを配置し、**ただの int4 より情報の損失を抑える**ように設計されています。

- bitsandbytes では `bnb_4bit_quant_type="nf4"` で指定します
- **二重量子化（double quantization）** を併用すると、量子化に使う補助情報自体もさらに圧縮でき、メモリを追加で節約できます（`bnb_4bit_use_double_quant=True`）

## 3. LoRA とは何か

**LoRA（Low-Rank Adaptation）** は、巨大なモデルの重みを**そのまま凍結**し、各層に**小さな追加行列（アダプタ）だけを学習**する微調整手法です。

- 学習するパラメータが**ごく一部**で済むため、メモリ・時間・保存容量が小さい
- 元モデルは触らないので、**アダプタを差し替える**だけで用途を切り替えられる

## 4. QLoRA = NF4 + LoRA

**QLoRA** は、**NF4で4bit量子化して凍結したモデル**の上に、**LoRAアダプタを載せて微調整**する手法です。

```
[元モデル(16bit)] --NF4で4bit量子化--> [凍結された軽い土台] + [LoRAアダプタ(学習対象)]
                                                              ↑ ここだけ学習する
```

これにより、本来は大きなGPUが要る微調整を、**無料T4クラスのGPUでも**現実的に回せます。

!!! warning "混同に注意：「量子化」と「微調整」は別の話"
    初学者がいちばん混乱するポイントです。切り分けて覚えてください。

    - **量子化（NF4で4bitロード）= モデルを軽くして"動かす"こと**。微調整しなくても、推論を軽くするだけにも使えます。
    - **QLoRA = その軽くしたモデルを"鍛える（微調整する）"こと**。
    - つまり「4bitでロードした」＝「微調整した」ではありません。**ロード（軽量化）と微調整（学習）は独立した別ステップ**です。

## 5. いつ使う／いつ使わない

### 使うと良いとき
- 手元のGPUが小さく、**そのままでは載らない/学習できない**モデルを微調整したい
- 自分のデータでモデルの**ふるまいを少し寄せたい**（口調・形式・ドメイン適応など）

### QLoRA を「使わない」判断（実務で重要）
- **十分なGPUがあり最高精度を狙う**なら、フル微調整（全パラメータ学習）の方が良い場合がある
- **微調整は不要で、推論を軽く速くしたいだけ**なら、別アプローチが適する。例：**GGUF / llama.cpp**（CPUでも動かしやすい推論向け形式）。※GGUFは **第2章**で扱います（ここでは名前だけ覚えればOK）
- そもそも**プロンプトの工夫だけ**で要件を満たせるなら、微調整しない方が速くて安い

## コラム：モデルを選ぶときの安全チェック

「このモデル、安全に使える?」——特に提供元が海外（中国系など）のモデルだと不安になります。ですが、**怖がるべきは"国籍"ではなく"運用"**です。攻撃の入口を理解すれば、出自に関わらず安全に扱えます。

!!! info "まず大前提"
    - モデルの**重みはただの数値**で、それ自体は勝手にコードを実行しません。
    - Colab上の推論は**ローカル処理**です。あなたのコードが明示的に通信しない限り、会話が外部へ送られることはありません。

危険が生まれるのは、ほぼ次の経路だけです。**ロード前に下のチェックを通せばOK**。

| チェック | なぜ重要か | 安全な状態 |
| :-- | :-- | :-- |
| **safetensors 形式か** | 旧来の `.bin`（pickle）は**読み込み時に任意コードを実行**しうる | safetensors を使う |
| **`trust_remote_code` が不要か** | `True` だと**モデル作者のPythonを実行**する | 不要なモデルを選ぶ（標準対応モデル） |
| **公式org から取得しているか** | 偽フォークに毒が仕込まれることがある | 公式アカウントのモデルを使う |
| **ライセンス・ゲート** | 商用可否や再配布の条件、ログイン要否 | Apache-2.0 / MIT・非ゲート |
| **（用途次第）検閲・バイアス** | 提供元の規制を反映し、特定話題で偏ることがある | 出力品質が重要な用途のみ要確認 |

本教材の既定モデル **`HuggingFaceTB/SmolLM2-1.7B`（Apache-2.0・非ゲート・safetensors・`trust_remote_code` 不要）** は、この5項目をすべて満たします。
このチェックを通せば、たとえば `Qwen/Qwen2.5-1.5B`（中国・Alibaba製だが同じく Apache-2.0／safetensors／非ゲート）も**技術的には同様に安全**に扱えます。最初の「中華系だから不安」は、こうして**確認可能な基準に置き換える**のが実務の作法です。

## トラブルシュート（実行中に詰まったら）

ハンズオン中によく出る問題と対処です。**実行しながら別タブで参照**してください。

| 症状 | 主な原因 | 対処 |
| :-- | :-- | :-- |
| `CUDA out of memory (OOM)` | モデル/バッチが大きすぎ | より小さいモデル（1〜1.5B）に変える／バッチサイズや最大長を下げる／二重量子化を有効化 |
| セッションが切れて状態が消えた | Colab無料枠の切断 | 再接続し、**セットアップセルから順に流し直す**（Run all） |
| モデルのロードで `gated`/認可エラー | ゲート付きモデルを指定した | **非ゲートの Apache-2.0/MIT モデル**に差し替える（[公式リンク集](references.md)参照） |
| 別モデルに変えたら推論がおかしい | チャットテンプレート/トークナイザの差 | そのモデル付属のテンプレート・トークナイザを使う。まずは教材の既定モデルで通してから差し替える |
| `nvidia-smi` が出ない | GPU未割り当て | ランタイムのタイプを **T4 GPU** に変更（[環境構築](../../getting-started/environment-setup.md)参照） |
