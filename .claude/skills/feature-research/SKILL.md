# Feature Research（機能実装の調査・戦略立案）

アプリ機能（例：DeviceActivity 集計値同期、サブスク自動検出、Prisma 冪等同期、App Router API）や
Claude Code の制御・運用設計など、**「どう実装・実現するか」を詳細に調査し、実装戦略まで立案する**ためのスキル。
出典付きの外部調査と、このリポジトリの既存規約への当てはめ、独立視点での反証までを定型フローで回す。

---

## 0. 不変条件（最優先・常に遵守）

- **このスキルは調査・戦略立案のみを行う。アプリのコードを実装・変更・追加・削除・コミットしない。**
- 差分（diff）や具体的な実装コードは書かない。書くのは「実装戦略（どう作るべきか）」まで。
- **`verify` / `run` のような実アプリ起動・動作検証はスコープ外**。必要ならユーザーが別途実施する。
- 成果物は「出典付き調査レポート＋実装戦略（コード変更なし）」に限定する。
- 実装に着手する段階になったら、このスキルを終了し、別途 root `.steering/[YYYYMMDD]-[機能名]/` を正式に切る（このスキルはそこまで行わない）。
- `CLAUDE.md` のペルソナ運用・PII/機微データ方針・ローカルファースト・しきい値外出し方針は、本スキルでも常に優先する。

---

## 1. 入力（調査対象の確定）

1. **機能名あり**（例：`/feature-research DeviceActivity 集計値同期`）→ それを調査スコープとして確定し、フロー開始。
2. **機能名なし**で起動 → 「どの機能/テーマの実装方法を調査しますか？」と**1問だけ**確認する。必要なら `docs/functional-design.md` の未実装機能を選択肢として提示してよい。
3. **対象が曖昧／広すぎる**（例「サブスク管理全般」）→ 調査可能な**単一スコープ**へ絞るよう促す（例「サブスク自動検出ロジック」）。

確定した対象を「機能名スラッグ」（英小文字・ハイフン）に正規化し、成果物ディレクトリ名に使う。

---

## 2. 事前ガード（Step 0）

フロー開始前に必ず実施：

1. 上記「不変条件」を自分に再宣言する（実装しない）。
2. **PII ガード確認**（§7）。
3. **codex 実働可否を確認**：`codex login status` 相当で `Logged in` か判定。未ログインなら §8 の縮退へ分岐。
4. 対象機能に応じた**最適なペルソナ**を選ぶ（例：DeviceActivity → Swift/SwiftUI、同期 API → Next.js + Prisma + Zod、制御系 → AI開発オーケストレーター）。

---

## 3. 実行フロー（5ステップ・逐次パイプライン）

> 各ステップは前段の出力を入力に取る。**全ステップで実装・コミットを禁止**。

### Step 1 — 外部仕様・先行事例の調査（`deep-research`）
- **目的**：対象の公式仕様・実装パターン・既知の落とし穴を、出典付きで収集・反証検証する。
- **入力**：正規化した機能スコープ＋着眼点（API 制約、ライブラリ選定、セキュリティ/PII 影響、パフォーマンス）。
- **出力**：出典付き要約（主要な実装アプローチ候補、制約、リスク、参考リンク）。
- **ガード**：検索クエリ・取得内容に実 PII・資格情報・社外秘を含めない。実データを外部送信しない。

### Step 2 — 既存コードの規約・類似実装の把握（`Explore`）
- **目的**：このリポジトリの命名規約・レイヤ構成・類似実装を把握し、「既にある書き方」に合わせる土台を作る（read-only）。
- **入力**：機能スコープ＋Step 1 の技術キーワード。
- **対象例**：`apps/web/`（App Router / Prisma / Zod / `src/domain/scoring/` / `src/config/`）、`docs/`（functional-design・architecture・glossary・development-guidelines）。
- **出力**：関連ファイル・既存パターン・準拠すべき規約の一覧。

### Step 3 — このリポジトリ向けの実装戦略（`Plan`）
- **目的**：Step 1（外部）× Step 2（自コード）を統合し、**実装はせずに**実装戦略へ落とす。
- **出力の骨子**：
  - 採用アプローチと**却下した代替案＋理由**（トレードオフ）。
  - 変更が想定される重要ファイル／新規モジュール（**提案のみ。差分・コードは書かない**）。
  - データモデル・API・型・Zod 境界への影響。
  - PII / ローカルファースト / しきい値外出し方針との整合。
  - リスク・未確定事項・検証方法（後続で `verify`/`run` を使う想定だと明記。ただし本スキルでは実行しない）。

### Step 4 — 反証（`codex:rescue`）
- **目的**：Step 3 の戦略を独立した第二視点で**反証・別案提示**し、一案偏重を防ぐ。
- **入力**：Step 3 の戦略要約と主要な判断（採用アプローチ・前提）。
- **出力**：戦略の穴・別アプローチ・見落とし。妥当な指摘は戦略文へ反映する（差分ではなく戦略の更新）。
- **縮退**：codex 未ログイン時はスキップし、レポートに「反証未実施（理由）／信頼度留保」と明記（§8）。

### Step 5 — 記録（`knowledge-scribe`）
- **目的**：確定した実装パターン・採用理由・却下した代替案を記録し、次回調査で再利用可能にする。
- **出力**：§6 のフォーマットで調査レポートを保存する。

---

## 4. 成果物の保存先

```
research/[YYYYMMDD]-[機能名スラッグ]/investigation.md
```
- **プロジェクト直下の専用ディレクトリ `research/`** に格納する（アプリ機能・Claude Code 制御など横断的内容のため `.claude/` に閉じない）。
- ステアリング（`.steering/`）とは別系統。1調査=1サブディレクトリ。
- `[YYYYMMDD]` は**ユーザー提示の現在日付**を使う（不明なら確認する。日付を推測しない）。
- `research/` が無ければ作成する。

---

## 5. investigation.md フォーマット（雛形）

```markdown
# [機能名] 実装方法の調査・戦略（実装なし）
> 調査日：YYYY-MM-DD / 対象スコープ：… / 起用ペルソナ：… / codex反証：実施 or スキップ(理由)

## 1. 調査サマリ（結論）
## 2. 外部仕様・先行事例（出典付き）   ← Step1
## 3. 既存コードの規約・類似実装        ← Step2
## 4. 実装戦略（採用案）                ← Step3
## 5. 却下した代替案とトレードオフ      ← Step3
## 6. 反証で出た論点と対応              ← Step4
## 7. リスク・未確定事項・検証方針      ← 後続作業への申し送り
## 8. 参考リンク（出典一覧）
```

---

## 6. PII・機微データガード（全ステップ共通）

- `CLAUDE.md` の PII・機微データ方針を遵守する。
- **実データ不参照**：本番 DB、`~/.codex`、`.env`、`secrets/`、実領収書メール、実利用ログ、実メールアドレス等を読まない／レポートに転記しない。
- **合成データのみ**：例示は合成（ダミー）データで行う。
- **外部送信の最小化**：`deep-research` の検索や `codex:rescue` への入力に、実 PII・資格情報・社外秘を含めない（一般的な技術情報のみ）。
- 触れざるを得ない機微情報が出たら、**作業を止めてユーザーに確認**する。

---

## 7. 縮退動作（依存が使えないとき）

- **codex 未ログイン／`codex:rescue` 不可**：Step 4 をスキップし、Step 1–3・5 は通常どおり実行。investigation.md に「反証未実施（理由）／信頼度留保」を明記し、完了報告で「`! codex login` 後に反証込みで再実行可能」と案内する。
- `deep-research` / `Explore` / `Plan` / `knowledge-scribe` のいずれかが不在の場合も、可能な範囲で実行し、欠落をレポートに明記する（サイレントに省略しない）。

---

## 8. 完了時の振る舞い

スキル完了時、ユーザーへ必ず以下を提示する：

- 「これは**調査・戦略立案**であり、**コード実装は行っていない**」旨。
- 成果物パス（`research/[YYYYMMDD]-[機能名スラッグ]/investigation.md`）。
- codex 反証の実施有無（スキップ時は理由と再実行方法）。
- 実装に進む場合の次アクション（root `.steering/[YYYYMMDD]-[機能名]/` を正式に切る／`verify`・`run` で実証）。ただし**着手はユーザー判断**であり、このスキルでは行わない。
